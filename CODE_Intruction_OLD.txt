Goal
====
Complete the missing GuitarSet bridge on the `score_hpt` side so that:

1. `pytorch/train_ddsp.py` can run Route 2 with `proxy.type=diffsynth_guitar`
2. `pytorch/train_proxy.py` can run Route 3 on guitar data without ground-truth velocity labels

Project terminology
===================
Use the following terminology consistently in new code and docs:

- `DiffSynth`
  - DDSP-based differentiable synthesizer
  - replacement for a black-box synthesizer

- `DiffProxy`
  - differentiable proxy of a black-box synthesizer
  - in this project this means SFProxy

- `synthesizer-input tensors`
  - tensors sent from `score_hpt` into `DiffSynth` or `DiffProxy`
  - prefer `synth_*` tensor names instead of `proxy_*`

Important compatibility note:
- existing legacy `proxy_*` tensor names must still work if they already appear in old HDF5 files or old scripts
- add aliases rather than breaking compatibility

Background
==========
The repo already has three routes wired inside `score_hpt`:

- Route 1: `pytorch/train.py`
- Route 2: `pytorch/train_ddsp.py`
- Route 3: `pytorch/train_proxy.py`

The DiffSynth-Guitar wrapper already exists in:

- `score_hpt/pytorch/proxy/ddsp_guitar.py`

The wrapper should consume these preferred batch keys:

- `synth_midi_pitch`
- `synth_string_index`
- `synth_midi_onsets`
- `synth_midi_activity`

Legacy aliases that should remain accepted:

- `proxy_midi_pitch`
- `proxy_string_index`
- `proxy_midi_onsets`
- `proxy_midi_activity`

The problem is that the current built-in dataset packers in `score_hpt/pytorch/data_generator.py` only support:

- `maestro`
- `maps`
- `smd`

So the score-side GuitarSet bridge is still missing.

B. Store both standard score_hpt targets and guitar synthesizer-input tensors
------------------------------------------------------------------------------
The HDF5 format for guitar should include:

1. waveform and standard `score_hpt` rolls for the student model
2. DiffSynth-Guitar synthesizer-input tensors for Route 2

Required stored content:

- `waveform`
  - mono mic audio

- standard score_hpt rolls at `cfg.feature.frames_per_second`
  - `onset_roll`        shape `[T_score, 88]`
  - `frame_roll`        shape `[T_score, 88]`
  - `offset_roll`       shape `[T_score, 88]`
  - `exframe_roll`      shape `[T_score, 88]`
  - `pedal_frame_roll`  shape `[T_score]` and it can be all zeros for guitar
  - `velocity_roll`     shape `[T_score, 88]`
  - `has_velocity_target=0`

Notes:
- guitar datasets usually do not provide reliable MIDI velocity labels
- it is acceptable to store `velocity_roll` as zeros if `has_velocity_target=0`
- Route 2 and Route 3 should then rely on differentiable supervision instead of label supervision

Required preferred synthesizer-input tensors for DiffSynth-Guitar:

- `synth_midi_pitch`    shape `[6, T_synth]`
- `synth_string_index`  shape `[6, T_synth]`
- `synth_midi_onsets`   shape `[6, T_synth]`
- `synth_midi_activity` shape `[6, T_synth]`

Legacy aliases may also be stored if that reduces migration pain:

- `proxy_midi_pitch`
- `proxy_string_index`
- `proxy_midi_onsets`
- `proxy_midi_activity`

Also store timing metadata explicitly:

- `synth_input_frame_rate`
- `score_frame_rate`
- `duration`
- `split`

C. Parse GuitarSet from JAMS
----------------------------
Use GuitarSet annotations from `annotation/*.jams`.

Expected local folder structure is the same one already used by `synthesizer/ddsp-guitar/data.py`:

- `annotation/*.jams`
- `audio_mono-mic/*_mic.wav`
- optionally `audio_hex-pickup_debleeded/*_hex_cln.wav`

Important:
- `score_hpt` only needs the mono mic waveform for the student input
- the hex pickup audio does not need to be used inside `score_hpt`
- parse per-string note events from the JAMS file
- use those note events to create:
  - 88-key score rolls for the student
  - 6-string synthesizer-input rolls for DiffSynth-Guitar

D. Make timing explicit in the DiffSynth-Guitar bridge
------------------------------------------------------
This is critical.

Do not assume the student frame rate, the GuitarSet stored frame rate, and the DiffSynth checkpoint frame rate are already identical.

The timing bridge must be explicit.

Required behaviour:

1. the dataset may store synthesizer-input tensors at a source frame rate
2. the wrapper must crop them using that source frame rate
3. the wrapper must resample them to the DiffSynth-Guitar model frame rate before rendering
4. the route must not rely on hidden defaults

Important naming for timing metadata:

- preferred batch key: `synth_input_frame_rate`
- legacy alias still accepted: `proxy_frame_rate`

Also make the supervision-side analysis parameters explicit where relevant:

- frame rate
- sample rate
- hop size / FFT parameters if they affect exported features

The main rule is simple:

- no hidden timing assumptions
- no hidden feature-rate assumptions
- no hidden audio-analysis assumptions

E. Keep weak-supervision training clean
---------------------------------------
For guitar training, the intended setup is usually:

- `dataset.has_velocity_target=false`
- `loss.supervised_weight=0.0`
- `loss.proxy_weight>0`

So make sure:

1. Route 2 can run with no GT velocity labels
2. Route 3 can run with no GT velocity labels
3. batches still provide `onset_roll` and `frame_roll` because they are needed by the score-informed model and by DiffProxy note construction
